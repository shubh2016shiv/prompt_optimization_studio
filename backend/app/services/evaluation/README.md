## 1. System Architecture Overview

### 1.1 The Quality Gate Concept

The `app.services.evaluation` package implements APOST's **internal quality gate**. It is not an exposed HTTP endpoint that a user calls directly. Instead, it is an integrated pipeline stage that evaluates and selectively modifies generated prompt variants before they are ever returned to the end user.

Regardless of whether the user requests an optimization using KERNEL, XML Structured, TextGrad, or Progressive Disclosure, the resulting variants are intercepted by the `PromptQualityCritic`. This ensures that every prompt generated by APOST passes a rigid, objective, and cross-framework standard of quality.

### 1.2 Data Flow into the Critic

The integration surface sits at the base strategy layer. Every concrete optimizer framework ends its `generate_variants()` routine by passing its intermediate outputs into the `_refine_variants_with_quality_critique()` helper method.

```text
┌────────────────────────────────────────────────────────┐
│             Optimization Framework Engine              │
│ (KERNEL, XML Structured, TextGrad, CoT Ensemble, etc.) │
└──────────────────────────┬─────────────────────────────┘
                           │ 3 Generated Prompt Variants
                           ▼
┌────────────────────────────────────────────────────────┐
│             BaseOptimizerStrategy Layer                │
│       _refine_variants_with_quality_critique()         │
└──────────────────────────┬─────────────────────────────┘
                           │ Parallel Async Execution
                           ▼
┌────────────────────────────────────────────────────────┐
│             PromptQualityCritic Service                │
│                                                        │
│  1. Critique: G-Eval Rubric Scoring (gpt-4.1-nano)     │
│  2. Gate Check: Is Score >= 70?                        │
│  3. Enhance: Surgical Rewrite if Score < 70            │
└──────────────────────────┬─────────────────────────────┘
                           │ Annotated & Enhanced Variants
                           ▼
┌────────────────────────────────────────────────────────┐
│                 API Response Models                    │
│   (PromptQualityEvaluation, VariantTCRTEScores)        │
└────────────────────────────────────────────────────────┘
```

The `PromptQualityCritic` handles one variant at a time and operates statelessly, leveraging `asyncio.gather` on the base strategy layer to process all three variants via parallel LLM requests.

## 2. G-Eval CoT Decomposition Mechanism

### 2.1 The Fixed Judge Strategy

Evaluation inside APOST requires an absolute baseline. If an Anthropic Claude model evaluates Claude prompts, and an OpenAI GPT model evaluates GPT prompts, the system becomes vulnerable to **self-evaluation bias**, where models heavily favor their natural structural tendencies over objective rigor.

To nullify this bias, the evaluation engine enforces a **Fixed Judge Model** policy defined centrally in `evaluation_rubric.py`. 
- **Judge Provider:** `openai`
- **Judge Model:** `gpt-4.1-nano`

By standardizing on a singular small, extremely fast, and instruction-adherent model, APOST guarantees that all frameworks and target providers are evaluated equally across a level playing field.

### 2.2 Decomposed Scoring Over Direct Rating

The core evaluation logic utilizes the **G-Eval (Generative Evaluation)** strategy. Standard grading prompts (e.g., "Rate this prompt from 1-100") consistently exhibit variance and poor human alignment. 

Instead of a direct score request, the APOST judge incorporates a **Chain-of-Thought Binary Checklist Decomposition**. For every dimension in the rubric, the judge must explicitly answer a specific set of binary questions regarding intent, format, and behavior before aggregating those truths into a localized integer score.

This forces the generation of an internal evaluation trace (`reasoning`), drastically reducing hallucinated scores and grounding the final numeric value in objectively verifiable text features.

## 3. The Seven-Dimension Quality Rubric

The evaluation framework abstracts structural readiness into seven discrete dimensions, internally tracked by the `DimensionScores` dataclass in `critique_result.py`.

### 3.1 Dimension Breakdown and Weighting

The first five dimensions map analogously to the TCRTE gap analysis system, while the final two address generation-specific performance metrics. Some dimensions execute a higher correlation with downstream production success and are consequently granted a higher aggregate weight.

| Dimension | Description | Weight |
|---|---|---|
| **Role Clarity** | Defines who the model is, its expertise level, and the negative bounds of its persona scope. | **1.0x** |
| **Task Specificity** | Defines an unambiguous core action, specifies explicit measurable success criteria, and borders scope. | **1.5x** |
| **Constraint Completeness** | Captures the existence of strict positive, negative, and verifiable output limits. | **1.5x** |
| **Output Format** | Validates exact structure specification, JSON schema validation, or required headers. | **1.0x** |
| **Hallucination Resistance** | Checks for context isolation boundaries, uncertainty instructions ("I don't know"), and anti-fabrication locks. | **1.0x** |
| **Edge Case Handling** | Tests operational behavior for missing input, ambiguous data, and fallback formats. | **1.0x** |
| **Improvement Over Raw** | A comparative metric confirming structural capability improvements without intent distortion against the raw user input. | **1.0x** |

## 4. Scoring and Evaluation Thresholds

### 4.1 Weighted Overall Calculation

The `PromptQualityCritic` calculates the `overall_score` via a purely mathematical weighted average to prevent the LLM from arbitrarily hallucinating final totals out of alignment with its decomposed arrays.

```text
overall_score = round( Σ (dimension_score × weight) / Σ weights )
```

The resulting `overall_score` dictates the fate of the variant. APOST sets the `QUALITY_GATE_THRESHOLD` at a strict `70`. Any prompt achieving this threshold naturally passes entirely untouched to the user.

### 4.2 Grade Boundaries

For UI representation and broad categorical communication, scores map to standardized grade thresholds:
- **A** (90-100) — Exceptional coverage
- **B** (80-89) — Production-ready coverage
- **C** (70-79) — Safe base coverage (Passes Quality Gate)
- **D** (50-69) — Incomplete or ambiguous (Triggers Enhancement)
- **F** (0-49) — Functionally invalid

## 5. The Automated Enhancement Pipeline

### 5.1 Surgical Injection vs. Broad Rewrite

When a variant scores **below 70**, it triggers the `enhance_prompt_from_critique()` method. 

The most common failure mode of chained LLM modifications is intent degradation—where the second pass erases the nuances of the first. APOST mitigates this by restricting the enhancement model from restructuring the prompt. The `ENHANCEMENT_SYSTEM_PROMPT` enforces strict **surgical injection**, accepting the exact list of `weaknesses` and `enhancement_suggestions` generated by the critique and splicing the necessary text into the most logical segment of the existing structure.

### 5.2 Enhancement Decision Tree

```text
                     ┌──────────────────┐
                     │ CritiqueResult   │
                     │ (overall_score)  │
                     └────────┬─────────┘
                              │
                              ▼
                     ┌────────────────┐
               Yes   │ score >= 70 ?  │   No
          ┌──────────│ (Quality Gate) │──────────┐
          │          └────────────────┘          │
          ▼                                      ▼
 ┌────────────────┐                   ┌───────────────────────────┐
 │ Skip Enhance   │                   │ enhance_prompt_from_      │
 │ final = draft  │                   │ critique()                │
 └────────────────┘                   └────────────┬──────────────┘
                                                   │
                                                   ▼
                                      ┌───────────────────────────┐
                                      │ Safety Validations:       │
                                      │ Length > 50% original?    │
                                      │ Did it avoid truncation?  │
                                      └────────────┬──────────────┘
                                                   │
                                                   ▼
                                      ┌───────────────────────────┐
                                      │ Return Enhanced Prompt    │
                                      │ (Or fallback to draft on  │
                                      │ validation failure)       │
                                      └───────────────────────────┘
```

If the LLM enhancer drops syntax tokens, outputs an empty string, or catastrophically reduces the file length (less than 50% length of the original), the pipeline catches it. It logs an anomaly warning and safely reverts to emitting the original, unenhanced version, preserving content over structure.

## 6. Resilience and Graceful Degradation

Because `PromptQualityCritic` forces secondary and tertiary LLM network calls directly inside the critical path of the standard user generation loop, resilience protocols are paramount.

### 6.1 Fallback Mechanics

The system operates under the inviolable rule: **The quality gate must never block the end-user.**

- **Critique API Faults:** If the call to `gpt-4.1-nano` encounters an API timeout, rate limit, or catastrophic network failure, the system intercepts the exception and issues a `_create_default_passing_result()`. This defaults every dimension score to exactly `70`, registers a synthetic empty weakness array, and forces the `passed_quality_gate` to Boolean true, ensuring progress.
- **Malformed JSON Faults:** If the judge outputs arbitrary non-JSON text or all dimension responses are interpreted strictly as `0`, the logic assumes JSON parsing failure and equally defaults to the passing result.
- **Enhancement API Faults:** If the enhancement loop fails execution, the system maintains the `CritiqueResult` metadata correctly but returns the original unmodified prompt.

## 7. API Boundaries and Mappings

### 7.1 Mapping Internal Models to API Responses

Internally, the `PromptQualityCritic` relies entirely on pure Python dataclasses (`CritiqueResult`, `DimensionScores` residing in `critique_result.py`). It never operates directly on standard API Pydantic models to ensure pure stateless computation and light footprint initializations.

At the threshold crossing where control returns to the base strategy layer, internal measurements are hydrated into API standards (`PromptQualityEvaluation`, `VariantTCRTEScores`).

The UI explicitly showcases the foundational TCRTE scores across the final variants. To prevent UI breakage, the backend accurately collapses the complex seven-dimension rubric into the familiar five-dimension system before sending:

| Rubric Internal Dimension | UI Target Field |
| ------------------------- | --------------- |
| `task_specificity`        | `task`          |
| `constraint_completeness` | `context`       |
| `role_clarity`            | `role`          |
| `edge_case_handling`      | `tone`          |
| `output_format`           | `execution`     |

*Note: `improvement_over_raw` and `hallucination_resistance` remain explicitly available in the `PromptQualityEvaluation.dimensions` object payload but are abstracted from the smaller TCRTE summary bar graphics.*
